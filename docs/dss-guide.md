# Руководство по СППР: как это работает

Ситуационный центр — это **система поддержки принятия решений (СППР / DSS)**, а не
просто мониторинг. Она ведёт оператора по полному циклу принятия решения:

```
OBSERVE → ORIENT → PROJECT → DECIDE → ACT → LEARN
 показа-   ситуа-   прогноз   рекомен-  регла-  исход +
 тели/     ция +    + упреж-  дация     мент    дообучение
 коридор   причина  дение     (NBA)             (win-rate)
```

Каждый узел реализован отдельным модулем; ниже — что это, как пользоваться через
**кокпит** (UI) и через **API**.

## 2. Понятийная модель

| Сущность | Что это | Модуль |
|---|---|---|
| **Цель (Goal)** | бизнес-цель, за которую кто-то отвечает | M2 |
| **Показатель (Indicator)** | измеримая величина с **целевым коридором** `[low, high]` | M2 |
| **Фактор (Factor)** | что влияет на показатель; связывает его с метриками | M2 |
| **Отклонение (Deviation)** | выход показателя за коридор; «хроника» = держится N периодов | M3 |
| **Ситуация (Situation)** | коррелированные отклонения + влияние + первопричина | M4 |
| **Предиктивный алерт** | прогноз выйдет за коридор → упреждение ДО пробоя | M5 |
| **Playbook** | правило «при таком отклонении — делать так» (+ регламент) | M7 |
| **Рекомендация (NBA)** | ранжированная альтернатива действия | M7 |
| **Процесс (Process)** | исполняемый регламент: шаги, чек-листы, отчёты | M8 |
| **Решение + Исход** | принятая рекомендация и сработала ли она | M10 |
| **Сценарий (What-if)** | «что если изменить показатель X» + оценка потенциала | M6 |

Иерархия: **Цель → Показатель → Фактор → Метрика**. Показатель «загорается» когда
его значение (взвешенное среднее метрик его факторов за последние ~5 минут) выходит
за коридор.

## 3. Сценарий работы (end-to-end)

### Шаг 1. Завести показатель с коридором (M2)
Кокпит пока редактирует показатели через API/Swagger (UI-формы — на будущее).

```http
POST /api/v1/indicators/goals      { "name": "Доступность ЦОД", "owner_role": "ops" }
POST /api/v1/indicators/           {
  "name": "Сеть ядра ЦОД", "unit": "%", "goal_id": "<goal>",
  "target_low": 0, "target_high": 100, "direction": "above",
  "chronicle_threshold": 2,
  "factors": [{ "name": "потери", "metrics": ["core_packet_loss"] }]
}
```
- `direction`: `both` | `below` | `above` — какая сторона выхода считается отклонением.
- `chronicle_threshold`: на каком подряд идущем периоде отклонение становится
  «хроникой» (повышается до `critical` и **авто-генерируются рекомендации**).
- Подписки: `POST /api/v1/indicators/{id}/subscriptions`.

### Шаг 2. Наблюдать отклонения и ситуации (M3/M4) — вкладка «Обзор»
- **Дерево показателей** со светофором: 🟢 в коридоре · 🟠 прогноз пробоя · 🔴 пробой · ⚪ неактивен.
- **Активные ситуации** — таблица по убыванию `impact`. Клик по строке → **drawer**:
  гипотеза первопричины, связанные отклонения, панель рекомендаций.
- Кнопка **«Коррелировать»** запускает группировку отклонений в ситуации немедленно
  (иначе это делает фоновая задача каждые 3 минуты).

Движок: показатели оцениваются Celery-задачей `evaluate_indicators_task` каждые 2 мин;
отклонения и «хроника» пишутся в `deviations` / `chronicles`. Корреляция использует
**граф зависимостей** между показателями (`POST /api/v1/situations/dependencies`):
два отклонения попадают в одну ситуацию, если их показатели связаны в графе и
случились в одном временном окне. Первопричина = верхний по графу пробитый показатель.

### Шаг 3. Получить рекомендацию и принять решение (M7) — drawer ситуации
1. Завести **playbook** (база знаний):
   ```http
   POST /api/v1/playbooks {
     "name": "Сетевой сбой → резерв", "trigger_severity": "critical",
     "trigger_direction": "above", "effect_score": 3.0,
     "process_template_id": "<template>", "indicator_ids": ["<indicator>"],
     "actions": [{ "action": "переключить на резерв", "checklist": ["уведомить дежурного"] }]
   }
   ```
2. В drawer ситуации нажать **«Сгенерировать рекомендации»** → ранжированный список
   (по эффекту × severity × устойчивости × специфичности × **win-rate**).
   При «хронике» рекомендации появляются **сами** (M3→M7).
3. **«Принять»** у топ-рекомендации → если у playbook привязан регламент, **запускается
   процесс** (M8), привязанный к отклонению; остальные альтернативы → `dismissed`.
   Это и есть фиксация решения.

### Шаг 4. Исполнить регламент (M8) — вкладка «Процессы»
- Список инстансов процессов; клик → **drawer с таймлайном шагов**.
- **Волновая модель**: шаги с одинаковым `step_order` идут параллельно; разные порядки —
  последовательно. Волна сдвигается, когда все шаги текущей волны завершены.
- По активному шагу: **Начать → отметить чек-лист → Завершить** (можно «без чек-листа»).
- Просроченные по SLA шаги помечаются (фоновая задача `check_process_step_sla_task`, 5 мин).

Шаблоны: `POST /api/v1/processes/templates` (шаги, роли, чек-листы, `due_after_minutes`).

### Шаг 5. Упреждение (M5) — вкладка «Предиктив»
- Таблица предиктивных алертов (прогноз выйдет за коридор, ETA, уверенность high/medium).
- Клик по строке → **график прогноза с коридором** (линия `yhat` + доверительный
  интервал + границы коридора). Можно подтвердить/закрыть.
- Движок (`predict_indicators_task`, 15 мин) строит прогноз (Prophet) по показателям с
  одной метрикой и поднимает алерт, если **край доверительного интервала** пересечёт
  коридор (раннее предупреждение, `medium`) или сам прогноз (`high`).

### Шаг 6. Журнал решений и обучение (M10) — вкладка «Решения»
- **Журнал решений**: принятые рекомендации + исход. У решения без исхода — кнопки
  **«Устранено / Не устранено»**. Авто-исход выводит фоновая задача
  (`evaluate_decision_outcomes_task`, 10 мин): процесс завершён + отклонение закрыто =
  успех.
- **Эффективность playbook'ов**: win-rate (доля успешных решений) прогресс-баром.
  Win-rate **возвращается в скоринг M7** — успешные playbook'и ранжируются выше
  (после ≥3 исходов). Это замыкает контур **Learn → Decide**.

### Шаг 7. What-if (M6) — вкладка «What-if»
- Список сценариев с «потенциалом». **«Новый сценарий»** → допущения
  «показатель → `target` / `delta` / `delta %` → значение».
- **«Прогнать»** → текущее vs проекция, эффект (улучшение/ухудшение), **потенциал** =
  избежанный impact (severity × влияние вниз по графу зависимостей). Помогает сравнить
  «а что если предпринять действие X».

## 4. Кокпит — карта вкладок

| Вкладка | Модуль | Что делает оператор |
|---|---|---|
| **Обзор** | M2/M3/M4 | дерево-светофор, активные ситуации, drawer с рекомендациями |
| **Предиктив** | M5 | предиктивные алерты + график прогноза с коридором |
| **Процессы** | M8 | таймлайн шагов, чек-листы, отчёты, эскалация |
| **Решения** | M10 | журнал решений, фиксация исхода, win-rate playbook'ов |
| **What-if** | M6 | конструктор сценариев + оценка потенциала |

## 5. Фоновые задачи (Celery beat)

| Задача | Период | Назначение |
|---|---|---|
| `evaluate_indicators_task` | 2 мин | оценка показателей против коридора → отклонения/хроника (M3) |
| `correlate_situations_task` | 3 мин | корреляция отклонений в ситуации (M4) |
| `predict_indicators_task` | 15 мин | прогноз → предиктивные алерты (M5) |
| `check_process_step_sla_task` | 5 мин | эскалация просроченных шагов процесса (M8) |
| `evaluate_decision_outcomes_task` | 10 мин | авто-исходы решений → win-rate (M10) |

Запуск воркера/бита: `celery -A tasks.celery_app worker -l INFO` и
`celery -A tasks.celery_app beat -l INFO`. Без них всё работает по явному вызову API
(кнопки «Коррелировать», «Сгенерировать», `/scenarios/{id}/run` и т.д.).

## 6. Карта API (префикс `/api/v1`)

| Область | Эндпоинты |
|---|---|
| Показатели/цели (M2) | `/indicators`, `/indicators/goals`, `/indicators/tree`, `/indicators/{id}/subscriptions` |
| Отклонения (M3) | `/deviations`, `/deviations/{id}/acknowledge|resolve`, `/deviations/chronicles/list` |
| Ситуации (M4) | `/situations`, `/situations/correlate`, `/situations/dependencies`, `/situations/{id}/status` |
| Предиктив (M5) | `/predictions`, `/predictions/run`, `/predictions/forecasts/{id}/latest` |
| Знания/рекомендации (M7) | `/playbooks`, `/recommendations/generate`, `/recommendations/{id}/accept|dismiss` |
| Процессы (M8) | `/processes/templates`, `/processes/instances`, `/processes/assignments/{id}/start|checklist|complete` |
| Решения (M10) | `/recommendations/decisions`, `/recommendations/{id}/outcome`, `/playbooks/{id}/stats` |
| What-if (M6) | `/scenarios`, `/scenarios/{id}/run` |

Аутентификация: `POST /token` (логин) ставит httpOnly-cookie; программно — Bearer-токен
из тела ответа. RBAC: права вида `read:indicators` / `write:situations` и т.д.; роль
`admin` обходит проверки. Всё мультитенантно (`tenant_id` в каждом запросе).
