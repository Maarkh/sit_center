# Целевая архитектура СППР (Decision Support System)

> Проектный документ. Сводит воедино: 6 функциональных требований заказчика,
> классическую теорию DSS (4 подсистемы Sprague/Turban), фазы принятия решения
> Г. Саймона (Intelligence → Design → Choice), модель ситуационной осведомлённости
> Endsley / цикл OODA, практические паттерны **ИСУ СберКорус**, и 8 выявленных
> пробелов текущей реализации.
>
> Статус: проект целевого состояния (To-Be). Сравнение с As-Is — в конце.
>
> **Реализовано (2026-06-03): Фаза 1 (M2+M3+M8 MVP) + M7 + M5 + M4.** Бэкенд: миграции
> 010–015, модели, API под `/api/v1/{indicators,deviations,processes,playbooks,recommendations,predictions,situations}`,
> Celery-задачи (коридор / SLA / предиктив / корреляция), движок рекомендаций, предиктивные
> алерты и корреляция отклонений в Ситуации (impact + первопричина). Контур охватывает
> Observe→Orient→Project→Decide→Act. 91 unit + 9 интеграционных тестов на реальной БД.
> Остаётся M10 (исходы), M6 (what-if) и UI кокпита (M11). Подробности — в разделе 6.

## 1. Концептуальная рамка

Система строится вокруг **замкнутого контура принятия решений** (OODA + фазы Саймона),
а не вокруг «алерт → уведомление». Каждый модуль закрывает свой участок цикла.

```
            ┌─────────────────────── контур обучения (Learn) ──────────────────────┐
            ↓                                                                       │
 [OBSERVE] ──→ [ORIENT] ──────→ [PROJECT] ──→ [DECIDE] ──────→ [ACT] ──→ [LEARN] ──┘
 Perception    Comprehension     Projection    Design+Choice    исполнение  исход
 (L1)          (L2)              (L3)          (Simon)          регламента  решения

 приём        корреляция в       прогноз        рекомендации     процесс с   решение→
 показателей  ситуацию, оценка   значений,      и сравнение      участниками результат,
 + коридор    влияния, причина   упреждение     альтернатив      и отчётами  дообучение
```

**Соответствие 4 подсистемам классической DSS:**

| Подсистема DSS | Модули целевой архитектуры |
|---|---|
| Data management | M1 Ingestion, M2 Indicator&Goal Model |
| Model management (MBMS) | M5 Forecasting, M6 Model & Scenario (what-if) |
| Knowledge base | M7 Knowledge & Recommendation |
| Dialog / UI | M11 Decision Cockpit |

## 2. Модульно-функциональная карта

12 функциональных модулей + сквозные сервисы. Для каждого: **назначение → вход → выход → фаза цикла**.

### M1. Ingestion & Time-Series (Observe / L1 Perception)
- **Назначение:** приём показателей из источников, нормализация, хранение time-series.
- **Вход:** push/pull метрики (REST, Kafka, Grafana webhook), внешние данные (рынок, погода, регуляторика).
- **Выход:** канонические записи в TimescaleDB.
- **Функции:** коннекторы источников, дедупликация, теги/измерения, ретеншн/компрессия.
- **As-Is:** ✅ есть (`canonical_metrics`, data-роуты, Kafka). Не хватает коннекторов внешних данных.

### M2. Indicator & Goal Model — иерархия показателей (Data subsystem)
- **Назначение:** связать сырые метрики с бизнес-целями. Ядро «привязки к KPI».
- **Модель (паттерн ИСУ):** `Метрика → Фактор → Показатель → Цель`.
- **Функции:**
  - реестр **показателей** с **целевым коридором** (нижняя/верхняя граница, статический ИЛИ динамический baseline по сезонности);
  - **подписки** (кто/какая роль следит за каким показателем) — закрывает п.1;
  - веса факторов, влияющих на показатель (модель влияния, L2).
- **As-Is:** 🟡 плоский реестр метрик + одиночные пороги. **Нет иерархии, коридора, подписок, целей.**

### M3. Deviation Detection & Chronicle (Observe → Orient)
- **Назначение:** обнаружение выхода за коридор и **устойчивости** отклонения.
- **Функции:**
  - оценка показателя против коридора (двусторонняя: ниже/выше) — закрывает п.2–3;
  - **«Хроника»** (паттерн ИСУ): отклонение, повторяющееся N периодов подряд → повышенный приоритет, снижение шума;
  - дедупликация, подавление, fingerprint.
- **As-Is:** 🟡 `rule_engine` сравнивает с одним порогом. **Нет коридора и хроники.**

### M4. Situation & Correlation — ситуация (Orient / L2 Comprehension)
- **Назначение:** превратить поток отдельных отклонений в **ситуацию** с пониманием смысла. Снимает «шум алертов».
- **Функции:**
  - корреляция связанных отклонений в одну **Ситуацию** (по времени, измерениям, графу зависимостей);
  - **оценка влияния** через модель зависимостей (показатель → затронутые объекты/процессы);
  - гипотеза **первопричины** (root cause).
- **As-Is:** 🔴 **нет.** Каждый алерт/инцидент изолирован.

### M5. Forecasting & Predictive Alerts (Project / L3 Projection)
- **Назначение:** прогноз значений и **упреждающее** предупреждение — закрывает п.6.
- **Функции:**
  - периодическая задача: строит прогноз (Prophet/др.) по подписанным показателям;
  - если `yhat`/доверительный интервал выйдет за коридор в горизонте N → **предиктивный алерт** «действуй заранее»;
  - горизонт и чувствительность настраиваются.
- **As-Is:** 🟡 прогноз есть, но **только по запросу** (`/forecasts/predict`); не замкнут в алерты.

### M6. Model & Scenario Management — MBMS / what-if (Model subsystem)
- **Назначение:** библиотека моделей и **сценарное моделирование** для ЛПР.
- **Функции:**
  - реестр моделей (прогнозные, оптимизационные, оценка эффекта);
  - **what-if**: «если предпринять действие X — как изменится показатель/прогноз»;
  - **оценка потенциала/эффекта** (паттерн ИСУ «Потенциал от внедрения»: денежная/целевая оценка выгоды устранения отклонения).
- **As-Is:** 🔴 **нет** (есть только ML-аномалии без выбора/комбинирования моделей и без what-if).

### M7. Knowledge Base & Recommendation (Design + Choice / Next-Best-Action)
- **Назначение:** превратить «что случилось» в «что делать». Это ядро *поддержки решения*.
- **Функции:**
  - **база знаний / playbook'и**: для типа отклонения — набор рекомендованных действий (паттерн ИСУ «наставничество», чек-листы);
  - **генерация альтернатив** (Simon Design) + **скоринг и приоритизация** по эффекту/потенциалу (Simon Choice; паттерн ИСУ «стакан задач» — топ-N по ценности);
  - **рекомендация Next-Best-Action** с обоснованием и уверенностью (explainability).
- **As-Is:** 🔴 **нет.** Система детектирует, но не рекомендует и не сравнивает варианты.

### M8. Process / Workflow Engine — исполняемый регламент (Act)
- **Назначение:** исполняемый процесс обработки отклонения с участниками и отчётами — закрывает п.4–5 (главная дыра).
- **Модель (паттерн ИСУ «Задача» + BPMN-подмножество):**
  - `ProcessTemplate` → `ProcessStep` (тип: sequential / parallel, условия перехода);
  - `ProcessInstance` (привязан к Ситуации/Инциденту);
  - `StepAssignment` (исполнитель/роль, **чек-лист**, статус, отчёт о выполнении, срок);
  - **полный след**: кто/какой шаг/когда/с каким результатом закрыл.
- **Функции:** запуск экземпляра по отклонению, активация шагов (послед./паралл.), отчёт участника («начать → чек-лист → завершить»), расчёт «процесс завершён», эскалация просроченных шагов.
- **As-Is:** 🔴 есть только `escalation_chains` (уведомить роль L1→L2→L3). **Нет шагов, назначений, чек-листов, отчётов, параллелизма.**

### M9. Notification & Escalation (Act, поддержка)
- **Назначение:** доставка уведомлений и эскалация по SLA.
- **As-Is:** ✅ есть (`notifications`, SLA, авто-эскалация, DLQ). Переиспользуется M8.

### M10. Decision Log & Learning Loop (Learn)
- **Назначение:** замкнуть контур: зафиксировать **принятое решение и его исход**, дообучать рекомендации.
- **Функции:**
  - журнал решений: ситуация → выбранная альтернатива → исполненный процесс → **результат** (устранено/нет, эффект);
  - обратная связь в M7 (какие playbook'и реально работают) и M3 (хроника как индикатор неустранения).
- **As-Is:** 🟡 есть `audit_log` (действия), но **нет связки «решение → исход» и обучения**.

### M11. Decision Cockpit (Dialog subsystem / UI)
- **Назначение:** интерактивный диалог ЛПР, а не просто дашборд.
- **Функции:** карта/обзор ситуаций, дерево «Показатель→Фактор→Метрика» со светофором (паттерн ИСУ), панель рекомендаций, **what-if-песочница**, лента задач и их статусов, прогнозные графики с коридором.
- **As-Is:** 🟡 есть React-дашборды, карта, графики. **Нет ситуационного кокпита, дерева показателей, панели рекомендаций, what-if.**

### Сквозные сервисы (Cross-cutting)
- AuthN/Z + RBAC, мультитенантность, аудит, наблюдаемость (Prometheus/OTel), интеграции (i-doit).
- **As-Is:** ✅ зрелые, переиспользуются всеми модулями.

## 3. Эскиз модели данных для новых модулей

```
# M2 Indicator & Goal Model
goals(id, name, tenant_id, owner_role, description)

           corridor_type[static|baseline], baseline_model_ref, tenant_id)
factors(id, indicator_id, name, weight)                 # влияет на показатель
factor_metrics(factor_id, metric_name)                  # метрика влияет на фактор
subscriptions(id, indicator_id, subscriber_role|user, channel, tenant_id)

# M3 Deviation & Chronicle

           detected_at, severity, status, fingerprint, tenant_id)

# M4 Situation
situations(id, title, root_cause_hypothesis, impact_score, status,
           opened_at, tenant_id)
situation_deviations(situation_id, deviation_id)        # корреляция
dependencies(src_object, dst_object, weight)            # граф влияния

# M5 Predictive
forecasts(id, indicator_id, dimensions, horizon_h, yhat, yhat_low, yhat_high,
          model_version, created_at)
predictive_alerts(id, indicator_id, breach_eta, projected_value, ...)

# M6 Model & Scenario
models(id, kind[forecast|optimize|impact], name, params, version)
scenarios(id, situation_id, name, assumptions_json, created_by)
scenario_results(scenario_id, model_id, projected_metrics_json, potential_value)

# M7 Knowledge & Recommendation
playbooks(id, deviation_type, name, description)
playbook_actions(id, playbook_id, order, action, checklist_json)
recommendations(id, situation_id, playbook_id, score, rationale, confidence)

# M8 Process / Workflow
process_templates(id, name, tenant_id)
process_steps(id, template_id, order, name, type[seq|parallel],
              assignee_role, checklist_json, due_after_min, transition_cond)
process_instances(id, template_id, situation_id|incident_id, status,
                  started_at, completed_at)
step_assignments(id, instance_id, step_id, assignee, status,
                 report_text, checklist_state_json, completed_at, due_at)

# M10 Decision Log
decisions(id, situation_id, chosen_recommendation_id, process_instance_id,
          decided_by, decided_at)
decision_outcomes(decision_id, resolved[bool], effect_value, evaluated_at)
```

Существующие `incidents`, `sla_policies`, `escalation_chains`, `alert_events`,
`audit_log`, `users/roles/tenants` — переиспользуются (инцидент = частный случай
Ситуации; SLA/эскалация подключаются к шагам процесса).

## 4. Поток данных (целевой, end-to-end)

```
M1 приём показателя
  → M2 сопоставление с показателем/коридором/целью
  → M3 отклонение? + хроника?
  → M4 корреляция в Ситуацию + оценка влияния + первопричина
  → M5 (параллельно) прогноз → предиктивный алерт ДО отклонения
  → M7 подобрать playbook → сгенерировать альтернативы → скоринг по эффекту (M6)
       → Рекомендация (Next-Best-Action)
  → ЛПР в M11 видит ситуацию + рекомендацию (+ may run what-if в M6)
  → M8 запуск процесса по выбранной альтернативе: шаги → участники → чек-листы → отчёты
  → M9 уведомления/эскалация по SLA на шагах
  → M10 фиксация решения и исхода → обратная связь в M7/M3
```

### Что переиспользуется как есть
Хранилище time-series, приём данных, оценка правил (как основа M3), уведомления,
SLA-дедлайны и эскалация, мультитенантность, RBAC, аудит, наблюдаемость, i-doit.
Это ~40% будущей системы — прочный фундамент **Observe** и поддержка **Act**.

### Принципиальный вывод
Текущая система — зрелый **мониторинг/алертинг** (фаза **Intelligence/Observe**).
Чтобы стать **СППР**, центр тяжести смещается в **Orient → Decide → Act → Learn**:
ключевые недостающие узлы — **M2 (иерархия показателей+коридор)**, **M7 (рекомендации)**
и **M8 (исполняемый процесс)**. ИСУ СберКорус подтверждает связку M2+M7+M8 как рабочую;
ваше преимущество над ИСУ — **M5 (real-time прогноз/упреждение)**, которого у ИСУ нет.

## 6. Дорожная карта (приоритеты по ценности/усилию)

**Фаза 1 — фундамент решения (даёт скачок ценности): ✅ РЕАЛИЗОВАНО (2026-06-03)**
1. ✅ **M2** иерархия Цель→Показатель→Фактор→Метрика + двусторонний коридор + подписки
   (`migration 010`, `api/routes/indicators.py`, `/api/v1/indicators` + `/tree`).
2. ✅ **M3** двусторонний коридор + хроника (`migration 011`, `core/deviation_engine.py`,
   `core.dss_tasks.evaluate_indicators_task` каждые 2 мин, `/api/v1/deviations`).
   Хроника: при `periods == chronicle_threshold` приоритет → critical + уведомление.
3. ✅ **M8 (MVP)** движок процессов: шаблон → шаги seq/parallel (волновая модель по
   `step_order`) → назначение → чек-лист → отчёт → таймлайн, эскалация просрочки по SLA
   (`migration 012`, `core/process_engine.py`, `/api/v1/processes`). Привязка к инциденту/отклонению.

> Покрытие: unit — `tests/test_{indicators,deviation_engine,process_engine}.py`;
> e2e — `tests/integration/test_dss.py` на реальной TimescaleDB (все 12 миграций
> накатываются чисто, deploy-path guard в CI расширен на DSS-таблицы).

**Фаза 2 — собственно «поддержка решения»:**
4. ✅ **M7** playbook'и + рекомендации + приоритизация по эффекту («стакан задач»)
   — РЕАЛИЗОВАНО (`migration 013`, `core/recommendation_engine.py`, `/api/v1/playbooks`
   + `/api/v1/recommendations`). Генерация ранжированных альтернатив для отклонения/инцидента,
   принятие → запуск регламента (M8) + журнал решения (proto-M10), сиблинги → dismissed.
5. ✅ **M5** замкнуть прогноз в предиктивные алерты (ваше отличие от ИСУ) — РЕАЛИЗОВАНО
   (`migration 014`, `core/predictive_engine.py`, `/api/v1/predictions`, beat-задача 15 мин).
   Прогноз (Prophet) проецируется на коридор: чистая `project_breach` берёт край дов. интервала
   как раннее предупреждение (medium), центральный yhat — как высокую уверенность (high).
6. **M10** журнал решений и исходов (частично закрыт M7: решение→процесс; остаётся **исход**).

**Фаза 3 — продвинутая СППР:**
7. ✅ **M4** корреляция в ситуации + модель влияния + root cause — РЕАЛИЗОВАНО
   (`migration 015`, `core/situation_engine.py`, `/api/v1/situations` + граф зависимостей,
   beat-задача 3 мин). Кластеризация отклонений (связность графа + временное окно),
   impact-score по влиянию, гипотеза первопричины = верхний по графу показатель.
8. **M6** what-if/сценарии + оценка потенциала.
9. 🟡 **M11** ситуационный кокпит — РЕАЛИЗОВАН (MVP): `/cockpit` (React) — KPI-карточки,
   дерево показателей со светофором (в коридоре / прогноз пробоя / пробой), таблица
   активных ситуаций, drawer ситуации с отклонениями и панелью рекомендаций
   (сгенерировать → принять = запуск регламента). Остаётся what-if-песочница (M6) и
   графики прогноза с коридором.

> Build-vs-buy для M8: при умеренной сложности процессов — собственный упрощённый
> движок на текущем стеке (Celery + БД + RBAC + audit). При сложных регламентах
> с согласованиями — рассмотреть внешний BPMN-движок (Camunda/Flowable/Temporal).
```
